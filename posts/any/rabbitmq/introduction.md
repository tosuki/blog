## The beginning, the concept of queue

A queue is a data structure that follows the First In First Out (FIFO) principle. It is a collection of elements that are added to the end of the queue and removed from the front of the queue. This means that the first element added to the queue will be the first one to be removed. It's structure used to stack things in a way that the first one to be added is the first one to be removed. This is useful in many scenarios, such as when you want to process tasks in the order they were received.

## How it may be implemented in computing?

There are many ways to implement a queue in computing, but one of the most common ways is to use a linked list. A linked list is a data structure that consists of a sequence of nodes, where each node contains a value and a reference to the next node in the list. This allows for efficient insertion and deletion of elements from the queue. Although this is good enough for monolith and isolated applications (that does not need to communicate to other services), it is not good enough for distributed systems, where you need to communicate between different services and applications. This is where message brokers come into play.

Message brokers uses the joy of networking (TCP/IP) to allow different applications to communicate with each other, even if they are running on different machines. They act as intermediaries that receive messages from one application and forward them to another application. This allows for decoupling of applications, which means that they can be developed and deployed independently of each other.

It's important to mention that although RabbitMQ uses network protocols, there are also queues that work using the operating system's memory, such as in-memory queues. These are faster than network-based queues, but they are not suitable for distributed systems, as they cannot be accessed by multiple applications running on different machines, and there is also the concept of queues in operating systems, such as the message queues in Linux, which are used for inter-process communication (IPC). These queues allow different processes to communicate with each other by sending messages through the queue.

**OBS:** The concept of sockets that we mention here is just the act of softwares that behaves as listeners/senders, which in the context of web, we are talking about web sockets. I'm saying this because there is also the concept of sockets in operating systems, which we call unix sockets for operating systems such as linux and macOS that are used for IPC (inter process communication). These sockets allow different processes to communicate with each other by sending messages through the socket.

## Queues built over network structures

When the queue is implemented over a network structure, it usually uses the TCP protocol as the transport layer. This means that the messages are sent over a network connection, and the queue is responsible for managing the connections and ensuring that the messages are delivered to the correct destination. This allows for distributed systems to communicate with each other, even if they are running on different machines. This is possible due to the consistency of the TCP protocol, which ensures that the messages are delivered in the correct order and that they are not lost during transmission. This is one of the reasons why RabbitMQ is a popular choice for message brokering, as it provides a reliable and efficient way to manage queues over a network.

**Funny fact:** The TCP channel is not the only way to implement a queue over a network structure, there are also other protocols such as UDP, which is a connectionless protocol that does not guarantee the delivery of messages. However, RabbitMQ uses TCP because it provides a reliable and efficient way to manage queues over a network, which is essential for distributed systems.

**Funny fact 2:** The start of the TCP connection is called the three-way handshake, which is a process that establishes a connection between two devices. The three-way handshake involves three steps: the client sends a SYN (synchronize) message to the server, the server responds with a SYN-ACK (synchronize-acknowledge) message, and the client responds with an ACK (acknowledge) message. This process ensures that both devices are ready to communicate and that the connection is established successfully and the HTTP protocol is the application protocol layer that is built on top of the TCP protocol, which means that it uses the TCP protocol to establish a connection and send messages between the client and the server and the web socket protocol uses HTTP to estabilish the inicial connection with the brokers.
